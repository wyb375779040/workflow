AI 产品新的工作方式，不是更快写 PRD，而是把协作链路整体前置和做实。

有了 Claude Code 之后，产品工作不应该再停留在“需求交接”。新的方式应该把情报、问题、规则、 技术共创、验收口径、数据复盘都放进同一条主线程里，让产品、设计、技术、数据围绕统一上下文持续协同。

产品名：产品技术ClaudeCode协作模式
适用对象：产品 / 设计 / 技术 / 数据 / 运营
PRD 不再是起点，而是阶段性产物。AI 也不只是写文档，而是前置分析、规则建模、研发协同、复盘沉淀的协作中枢。
Old vs New

旧工作方式 vs 新工作方式

过去是“交接式研发”，以后要变成“共创式研发”。先把问题和规则讲清楚，再进入文档、技术和开发。

旧方式
PRD → 技术评审 → 开发实现 → 联调验收
起点是写文档，很多隐性规则和边界条件被留到后面，返工往往集中在联调阶段暴露。
新方式
情报扫描 → 问题定义 → 功能规则 → 在线 PRD → 技术共创 → AI 辅助开发 → 联调验收 → 数据复盘 → 资产沉淀
PRD 变成共识文档，技术和设计提早介入，AI 贯穿前置分析到上线复盘。
维度 旧方式 新方式
工作起点产品写 PRD产品先做情报、目标、规则
协作模式串行交接多角色并行共创
AI 角色写文档 / 写代码工具知识中枢 + 分析助手 + 实现助手
技术介入时间PRD 完成后规则阶段提前介入
设计介入时间需求相对稳定后问题定义后尽早介入
返工来源联调才发现理解偏差前置在规则和验收阶段暴露问题
验收依据PRD 描述功能规则 + 验收清单 + 测试路径
产出沉淀文档散落统一目录、统一版本、统一上下文
Eight Stages

新的八阶段工作方式

整体链路从“先写 PRD”改成“先做判断、再建规则、再共创实现、最后复盘沉淀”。

1

情报扫描

在写 PRD 前先确认外部怎么做、内部哪里痛、机会值不值得做。

主责产品经理
AI 动作竞品拆解、反馈归类、数据摘要、机会点提炼
产出竞品分析、用户问题清单、机会判断卡
2

问题定义

把模糊想法变成明确的问题陈述，而不是直接开写大 PRD。

主责产品经理
AI 动作反向追问边界条件、识别隐性假设、补异常场景
产出问题定义卡、目标指标、范围边界、优先级
3

功能规则建模

在 PRD 之前先把规则讲清楚，把返工最多的部分前置。

主责产品经理
AI 动作生成规则草稿、状态流转、异常和权限分支
产出功能规则清单、状态流转、异常处理、权限矩阵
4

在线 PRD

把前面的情报、问题、规则，收敛成团队可以共享的正式文档。

主责产品经理
AI 动作生成结构化 PRD 初稿、补目录、整理变更记录
产出在线 PRD、版本记录、验收口径
5

技术共创与任务拆解

把“技术评审”从被动挑错，升级成产品、技术、AI 的方案共创。

主责技术负责人
AI 动作方案草稿、风险提示、开发拆解、测试点补充
产出技术方案卡、任务拆解、风险清单、联调依赖表
6

AI 辅助开发实现

让 Claude Code 进入研发主流程，而不是只在边缘帮忙。

主责技术负责人 / 开发
AI 动作生成代码初稿、补测试、做 review、解释上下文
产出可运行代码、测试脚本、自测记录
7

联调验收与上线

把“验收”从感受式判断，变成有清单、有路径、有结论的动作。

主责产品经理 + 技术负责人
AI 动作生成验收清单、测试路径、回归检查项、上线检查单
产出验收记录、问题单、上线检查单
8

数据复盘与资产沉淀

不是上线即结束，而是把经验回收到下一轮。

主责产品经理 / 数据
AI 动作汇总上线表现、识别漏斗问题、沉淀最佳实践和坑点
产出数据复盘、问题归因、AI 数据资产库
很多过去在联调时暴露的问题，本质上都不是开发问题，而是规则没有提前说清。
Internal Collaboration

产品内部怎么协同

新的工作方式不是让产品一个人更忙，而是让产品、设计、技术、数据围绕统一主线程协作。Claude Code 横向服务所有角色。

产品 Owner

定目标、控优先级、做关键决策。

功能 PM

拉通问题定义、功能规则、PRD 版本。

设计 / 体验

提前参与规则和关键路径设计，不再只接原型需求。

技术负责人

在规则阶段就介入，避免后置打回。

数据 / 运营

提前定义指标、验收口径、上线反馈闭环。

一个需求，一个主线程

  • 一个需求只保留一个主文档和一个主对话上下文
  • 所有变更围绕这条主线程展开
  • 不允许微信、口头、会议纪要各说一版

先规则，后长文

  • 没有规则清单，不进入 PRD
  • 没有异常路径，不进入开发
  • 没有验收清单，不进入上线

争议前置，不把问题堆到联调

  • 业务争议放在问题定义阶段解决
  • 技术争议放在技术共创阶段解决
  • 体验争议放在原型和关键路径阶段解决

AI 负责提效，人类负责拍板

  • AI 负责整理、补盲点、生成初稿、辅助执行
  • 产品、技术、设计仍然对自己的专业判断负责
三个必须统一的协同载体 作用
问题定义卡统一目标、范围、成功标准
功能规则清单统一系统行为和边界
验收上线清单统一完成定义和上线口径
AiPM Mapping

和 AiPM 五阶段的映射关系

新的工作方式不是推翻旧流程，而是把每个阶段做深、做实、做成可协同。

新工作方式阶段 AiPM 阶段归属
情报扫描 + 问题定义产品规划
功能规则 + 在线 PRD产品需求
原型补充 + 技术共创产品设计
上线验证 + 使用分析数据分析
复盘结论 + 方法沉淀Ai数据资产库

映射后的真实含义

  • 产品规划不再只是方向描述，还要前置情报扫描和问题定义。
  • 产品需求不再只是写 PRD，而要把规则、版本和验收口径讲清楚。
  • 产品设计不只交原型，还要参与关键路径、技术共创和体验判断。
  • 数据分析和资产沉淀不再是附属动作，而是下一轮迭代的起点。
Cadence

建议的实际运行节奏

一个标准需求可以按日推进。会议只保留真正需要人拍板的环节，其他同步优先通过 Claude Code 整理成异步文档。

时间 关键动作 主责
Day 1情报扫描 + 问题定义产品
Day 2功能规则评审产品 + 技术 + 设计
Day 3在线 PRD 定稿 + 技术共创产品 + 技术 + AI
Day 4-5Claude Code 辅助开发 + 自测技术
Day 6联调验收 + 补缺陷产品 + 技术
Day 7上线 + 数据观察点确认产品 + 技术 + 运营
Day 8+数据复盘 + 资产沉淀产品 + 数据 + AI

三类必须保留的会议

  1. 范围决策会：决定做什么、不做什么。
  2. 方案争议会：解决业务和技术冲突。
  3. 上线放行会：决定是否上线、是否回滚。
Pitfalls

落地时最容易踩的坑

这套工作方式最怕“看起来用了 AI，实际上还是老流程”。真正的变化在于是否把问题、规则、验收和复盘都纳入同一条主线程。

坑点 表现 纠偏方式
还是从写 PRD 开始文档很长，但规则不清楚强制先出问题定义卡和规则清单
AI 只做秘书只让 AI 写纪要，不让 AI 参与规则和实现把 AI 放进前置分析和开发流程
技术介入过晚PRD 写完后才发现不可做规则阶段就让技术进来
验收口径模糊联调时反复争论“算不算完成”验收清单跟 PRD 同步维护
上线后没人复盘每轮都在重复犯错固定数据复盘和资产沉淀动作
Conclusion

最终结论

Claude Code 时代，产品新的工作方式不是“更快写 PRD”，而是把产品工作从文档驱动，升级成问题驱动、规则驱动、协同驱动。

对内

  • 产品、设计、技术、数据可以围绕统一上下文持续共创。
  • 协作不再依赖碎片化会议和口头对齐，而是依赖统一主线程。
  • AI 不是边缘工具，而是贯穿全流程的效率中枢。

对外交付

  • 情报判断
  • 问题定义
  • 功能规则
  • 技术方案
  • 开发任务
  • 验收清单
  • 数据复盘
  • 方法资产
